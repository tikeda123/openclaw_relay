# アーキテクチャ概要

## 目的

OpenClaw の制御側 `OptionABC001` と worker 群の間に、第三ホスト上の Relay を置く。  
Relay は mailbox API で message を受け取り、対象 worker に配送し、返答を `OptionABC001` の human-facing `agent:main:main` session へ戻す。

## 配置

- Relay サーバ
  - `openclaw-relay.service`
  - health / UI / metrics: `127.0.0.1:18080`
  - state / audit / spool を保持
- `OptionABC001`
  - mailbox caller
  - 再注入先
- worker 群
  - `OptionDEF002` など
  - `/v1/responses` の配送先

## 主要コンポーネント

### mailbox API

`OptionABC001` / worker は `PUT /v1/messages` と `GET /v1/messages` で Relay とやり取りする。

### watcher

`watch_dir` 配下の JSON message を検出する。  
この経路は互換用の補助入力で、通常運用の正規経路ではない。

### message validation

schema、TTL、duplicate suppression、idempotency を検証する。

### dispatch

message の `to` に基づいて worker を解決し、worker 側 `/v1/responses` へ配送する。

### inject

worker から得た返答を `OptionABC001` の `returnSessionKey` に再注入する。  
`returnSessionKey` を省略した場合は `OptionABC001` の `default_session_key` を使う。現在の既定は human-facing `agent:main:main` で、互換のため `main` を指定した場合も同じ値に正規化する。

### audit / metrics / UI

- JSONL audit
- SQLite state
- `/metrics`
- 会話 UI `/`
- 運用監視 UI `/ops`

## データフロー

1. `OptionABC001` が Relay mailbox API に message を送る
2. Relay が message を validation し、`messages` に `RESERVED` で登録する
3. worker を解決して `/v1/responses` へ配送する
4. worker reply を `replies/*.json` と DB に保存する
5. `OptionABC001` の human-facing `agent:main:main` session に reply を再注入する
6. `DONE` または `DEADLETTER_*` に遷移する

## 運用上の確認ルール

- mailbox API の `from` は caller token から Relay が決める
- 返信確認は human-facing `agent:main:main` session を見る
- `inbox/` の有無で着信判定しない

RabbitMQ transport を使う場合:

- queue message は adapter / consumer が `ack` するまで broker 上に残る
- 受信した瞬間に自動削除はしない
- 失敗時は `nack(requeue=true)` で再配送する

## 状態遷移

主な message status:

- `RESERVED`
  - 受理済み、worker 配送待ち
- `FAILED_B`
  - worker 配送失敗、retry 待ち
- `B_REPLIED`
  - worker reply 取得済み、A 再注入待ち
- `FAILED_A_INJECTION`
  - A 再注入失敗、retry 待ち
- `DONE`
  - 往復完了
- `DEADLETTER_B`
  - worker 配送側で retry 枯渇
- `DEADLETTER_A`
  - A 再注入側で retry 枯渇

## endpoint とトンネル

- `endpoints.a`
  - 制御側 `OptionABC001`
- `workers.<name>`
  - worker 群
- `routing.default_worker`
  - `to` が一致しない場合のフォールバック

HTTP 接続は local tunnel 経由でも、直接 base URL でも構成できる。  
実運用では worker ごとに異なる。

- `OptionDEF002`
  - SSH tunnel 経由
- `OptionGHI003`
  - SSH tunnel 経由の direct worker adapter と mailbox worker を併用

worker 側 gateway が loopback bind なら SSH tunnel を使う。LAN bind で安全に到達できるなら direct base URL でもよい。

## 現状の制約

- source sync は `OptionABC001` 1 系統
- 会話 UI は Relay DB を主、session transcript を補助として使う
- OpenClaw 内部思考や tool call 全量は取らない
- worker は複数定義できるが、制御側 endpoint は `endpoints.a` 固定

## Human / Relay の分離

- 人間 -> `OptionABC001`
  - Telegram main session
- `OptionABC001` -> worker
  - Relay message transport
- worker -> `OptionABC001`
  - human-facing session
- `OptionABC001` -> 人間
  - worker 返答を整理して Telegram に返す

人間から見える会話は `agent:main:main` に統一する。Relay mailbox / RabbitMQ / injector は内部 transport として分離し、必要なら `notifyHuman=true` の reply だけを fixed-format で Telegram に direct notify する。

## Schema 方針

新規導入は `relay-message/v1` を使う。

- `conversationId`
- `messageId`
- `from`
- `to`
- `body`

が基本で、`task` は別 transport として扱わない。  
旧 `relay-envelope/v1` は互換のため受理するが、今後の運用説明では `message` を正とする。  
互換のため `fromGateway` / `toGateway` も引き続き受理する。
