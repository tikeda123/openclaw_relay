# アーキテクチャ概要

## 目的

OpenClaw の制御側 `OptionABC001` と worker 群の間に、第三ホスト上の Relay を置く。  
Relay は outbox の envelope を受け取り、対象 worker に配送し、返答を `OptionABC001` の session へ戻す。

## 配置

- Relay サーバ
  - `openclaw-relay.service`
  - health / UI / metrics: `127.0.0.1:18080`
  - state / audit / spool を保持
- `OptionABC001`
  - outbox source
  - 再注入先
- worker 群
  - `OptionDEF002` など
  - `/v1/responses` の配送先

## 主要コンポーネント

### source sync

`OptionABC001` の outbox を `rsync over SSH` で Relay の `watch_dir` へ同期する。

### watcher

`watch_dir` 配下の JSON envelope を検出する。

### envelope validation

schema、TTL、duplicate suppression、idempotency を検証する。

### dispatch

envelope の `toGateway` に基づいて worker を解決し、worker 側 `/v1/responses` へ配送する。

### inject

worker から得た返答を `OptionABC001` の `returnSessionKey` に再注入する。

### audit / metrics / UI

- JSONL audit
- SQLite state
- `/metrics`
- 会話 UI `/`
- 運用監視 UI `/ops`

## データフロー

1. `OptionABC001` が `outbox/pending/*.json` に envelope を置く
2. Relay が source sync で取り込む
3. watcher が envelope を検出する
4. Relay が envelope を validation し、`messages` に `RESERVED` で登録する
5. worker を解決して `/v1/responses` へ配送する
6. worker reply を `replies/*.json` と DB に保存する
7. `OptionABC001` の session に reply を再注入する
8. `DONE` または `DEADLETTER_*` に遷移する

## 状態遷移

主な message status:

- `RESERVED`
  - 受理済み、worker 配送待ち
- `FAILED_B`
  - worker 配送失敗、retry 待ち
- `B_REPLIED`
  - worker reply 取得済み、A 再注入待ち
- `FAILED_A_INJECTION`
  - A 再注入失敗、retry 待ち
- `DONE`
  - 往復完了
- `DEADLETTER_B`
  - worker 配送側で retry 枯渇
- `DEADLETTER_A`
  - A 再注入側で retry 枯渇

## ディレクトリ

主な runtime path:

- `var/inbox/pending`
  - source sync で入る watch 対象
- `var/lib/openclaw-relay/processing`
  - processing copy
- `var/archive`
  - duplicate / archive
- `var/deadletter`
  - deadletter artifact
- `var/lib/openclaw-relay/replies`
  - worker raw reply
- `var/lib/openclaw-relay/relay.db`
  - SQLite state
- `var/log/openclaw-relay/relay.log`
  - relay log
- `var/log/openclaw-relay/a2a-audit.jsonl`
  - audit log

## endpoint とトンネル

- `endpoints.a`
  - 制御側 `OptionABC001`
- `workers.<name>`
  - worker 群
- `routing.default_worker`
  - `toGateway` が一致しない場合のフォールバック

HTTP 接続は local tunnel 経由でも、直接 base URL でも構成できる。  
実運用では worker 側 gateway が `bind: loopback` のため、SSH tunnel を使う前提。

## 監視

UI と endpoint:

- 会話画面: `/`, `/ui`
- 運用監視画面: `/ops`, `/monitor`
- JSON: `/api/dashboard`
- health: `/healthz`
- ready: `/readyz`
- metrics: `/metrics`
- Alertmanager webhook: `/api/alertmanager/webhook`

## 現状の制約

- source sync は `OptionABC001` 1 系統
- 会話 UI は Relay DB を主、session transcript を補助として使う
- OpenClaw 内部思考や tool call 全量は取らない
- worker は複数定義できるが、制御側 endpoint は `endpoints.a` 固定
