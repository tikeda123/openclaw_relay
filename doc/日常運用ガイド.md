# 日常運用ガイド

## 目的

Relay を運用開始したあとに、日次確認、障害切り分け、再実行、保守を最短で回すための手順をまとめる。

前提:

- cross-node の通信は Relay mailbox API を正規経路とする
- OpenClaw 内部 subagent dispatch は、同一ノード内の補助用途として扱う
- 人間向けの見え方は `OptionABC001` の human-facing `agent:main:main` session に統一する
- Relay mailbox / RabbitMQ / injector は内部 transport として扱う
- Telegram direct notify は `notifyHuman=true` の依頼にだけ発火する

## 毎日見るもの

### 1. service

```bash
sudo systemctl status openclaw-relay.service --no-pager
sudo systemctl status openclaw-relay-cleanup.timer --no-pager
```

### 2. health / ready / metrics

```bash
curl -fsS http://127.0.0.1:18080/healthz
curl -fsS http://127.0.0.1:18080/readyz
curl -fsS http://127.0.0.1:18080/metrics
```

最低限の確認値:

- `healthz` が返る
- `readyz` が `{"status": "ok"}`
- `openclaw_relay_ready 1`
- `openclaw_relay_mailbox_queue_depth{mailbox="..."}` が滞留していない

### 3. UI

- 会話確認: `http://127.0.0.1:18080/`
- 監視確認: `http://127.0.0.1:18080/ops`

見るポイント:

- deadletter が増えていないか
- dispatch latency が急に悪化していないか
- tunnel / http が `healthy` のままか

## 日次の基本コマンド

### relay 状態を見る

```bash
PYTHONPATH=src python3 -m openclaw_relay --config config/relay.local.toml inspect --limit 20
```

worker 単位:

```bash
PYTHONPATH=src python3 -m openclaw_relay --config config/relay.local.toml inspect --worker WorkerGateway1 --limit 20
```

### control host から worker に送る

control host では mailbox script を標準手順にする。

```bash
relay-mailbox-send.sh \
  --to WorkerGateway1 \
  --body "調査を開始してください。"
```

公開 message 形式としては `from` / `to` / `body` を正とする。  
`from` は mailbox token から Relay が決める。

script をまだ入れていない場合の最小 API 例:

```bash
curl -X PUT http://127.0.0.1:18080/v1/messages \
  -H 'Authorization: Bearer '"$RELAY_MAILBOX_TOKEN_OPTIONABC001" \
  -H 'Content-Type: application/json' \
  -d '{"to":"WorkerGateway1","body":"調査を開始してください。"}'
```

運用ルール:

- cross-node 通信では OpenClaw 内部 subagent dispatch を使わない
- worker 返答は human-facing `agent:main:main` session に統一する
- `openclaw-relay emit` は使わない
- 詳細は [OptionABC001送信ガイド.md](OptionABC001送信ガイド.md)

### deadletter 一覧

```bash
PYTHONPATH=src python3 -m openclaw_relay --config config/relay.local.toml deadletters --limit 20
```

worker 単位:

```bash
PYTHONPATH=src python3 -m openclaw_relay --config config/relay.local.toml deadletters --worker WorkerGateway1 --limit 20
```

### deadletter replay

最新 1 件:

```bash
PYTHONPATH=src python3 -m openclaw_relay --config config/relay.local.toml replay-deadletter --latest
```

worker 単位:

```bash
PYTHONPATH=src python3 -m openclaw_relay --config config/relay.local.toml replay-deadletter --worker WorkerGateway1 --latest
```

## 障害時の確認順

1. `systemctl status openclaw-relay.service`
2. `curl /healthz`
3. `curl /readyz`
4. `curl /metrics`
5. `/ops`
6. `inspect --worker ...`
7. `deadletters --worker ...`
8. `journalctl -u openclaw-relay.service -n 100 --no-pager`

## 代表的な症状と対処

### `readyz` が失敗する

確認対象:

- control host / worker host の `responses` endpoint
- tunnel の local port 競合
- `known_hosts` 不整合
- mailbox token の未設定または不一致

### deadletter が増える

まず見るもの:

- `openclaw_relay_worker_attempt_total`
- `openclaw_relay_worker_deadletter_total`
- `journalctl`

よくある原因:

- worker 側 gateway down
- token 不整合
- `responses.enabled` が無効
- worker adapter down

### dispatch が遅い

見るもの:

- `openclaw_relay_worker_latency_seconds{stage="dispatch",stat="latest"}`
- worker 側の OpenClaw 応答時間
- Relay host と worker host 間の SSH / LAN 遅延

## 定期メンテナンス

### cleanup dry-run

```bash
PYTHONPATH=src python3 -m openclaw_relay --config config/relay.local.toml cleanup --older-than-days 7 --dry-run
```

### cleanup timer

```bash
sudo systemctl status openclaw-relay-cleanup.timer --no-pager
systemctl list-timers --all | grep openclaw-relay-cleanup
```

### ログ確認

```bash
journalctl -u openclaw-relay.service -n 100 --no-pager
tail -n 100 var/log/openclaw-relay/relay.log
tail -n 100 var/log/openclaw-relay/a2a-audit.jsonl
```

## 変更作業の手順

コード、設定、SSH、host 名解決を変えたら:

```bash
sudo systemctl restart openclaw-relay.service
sudo systemctl status openclaw-relay.service --no-pager
curl -fsS http://127.0.0.1:18080/readyz
```

## token ローテーション

現行実装:

- `token_env` があればそれを優先
- なければ tunnel 越しに remote `openclaw.json` から取得
- `401` / `403` が出たら token cache を破棄
- 次回 retry で再取得

つまり、remote token 更新時は多くのケースで Relay 再起動は不要。

## IP 変更時

IP が変わったら、まず Relay host 側の `/etc/hosts` または DNS を直す。

その後:

```bash
sudo systemctl restart openclaw-relay.service
curl -fsS http://127.0.0.1:18080/readyz
```

詳細は [IP変更時の対処.md](IP変更時の対処.md) を参照。
