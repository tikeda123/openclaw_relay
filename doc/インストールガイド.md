# インストールガイド

## 目的

OpenClaw の制御側 1 台と worker 1 台以上を、第三ホスト上の Relay でつなぐための初回導入手順をまとめる。

この文書は、まず 1 台の control host と 1 台の worker host を Relay に接続するところまでを対象にする。

## 推奨トポロジ

- Relay host
  - Linux 推奨
  - `openclaw_relay` を動かす
- Control host
  - `OptionABC001` 相当
  - outbox source と再注入先
- Worker host
  - `OptionDEF002` 相当
  - `/v1/responses` の配送先

## 前提

- control host と worker host に OpenClaw が導入済み
- Relay host から control host / worker host に SSH 接続できる
- control host / worker host で OpenClaw gateway が起動できる
- Relay host に Python `3.12+`, `git`, `rsync`, `ssh` client がある
- 名前解決は DNS か `/etc/hosts` を使う

補足:

- このリポジトリの実装は Relay host を別ホストに置く前提
- control host / worker host の OS は、SSH と OpenClaw が動けば Linux / macOS のどちらでもよい
- Windows はこのガイドでは扱わない

## 0. 使う名前を決める

まず 3 つのホスト名を決める。

- Relay host
  - 例: `relay-node`
- Control host
  - 例: `control-node`
- Worker host
  - 例: `worker-node`

以後の設定例ではこの 3 名を使う。

## 1. Relay host にリポジトリを置く

```bash
git clone <your-repo-url> openclaw_relay
cd openclaw_relay

python3 -m venv .venv
. .venv/bin/activate
pip install -e .
```

確認:

```bash
python3 --version
PYTHONPATH=src python3 -m unittest discover -s tests -v
```

## 2. Control host を準備する

### 2-1. outbox を作る

control host 上で OpenClaw workspace 配下に outbox を作る。

```bash
mkdir -p ~/.openclaw/workspace/outbox/pending
```

workspace が別なら、その実パスに読み替える。

### 2-2. `responses` endpoint を有効化する

`~/.openclaw/openclaw.json` で次を有効にする。

```json
"gateway": {
  "http": {
    "endpoints": {
      "responses": {
        "enabled": true
      }
    }
  }
}
```

反映後、OpenClaw gateway を再起動する。

### 2-3. SSH server を有効にする

Relay host から SSH で入れるようにする。  
macOS なら `Remote Login`、Linux なら `sshd` を有効にする。

## 3. Worker host を準備する

worker host でも、control host と同様に `responses` endpoint と SSH server を有効にする。

確認したいもの:

- OpenClaw gateway が起動している
- `gateway.http.endpoints.responses.enabled = true`
- Relay host から SSH で入れる

## 4. Relay host の名前解決を用意する

DNS がない場合は `/etc/hosts` を使う。

例:

```text
10.0.0.10 control-node
10.0.0.11 worker-node
10.0.0.12 relay-node
```

サンプルは [../config/relay.hosts.example](../config/relay.hosts.example) を参照。

確認:

```bash
getent hosts control-node worker-node relay-node
```

## 5. Relay host に SSH 鍵を作る

```bash
ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519_control -N ''
ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519_worker1 -N ''
```

作られた `.pub` を、それぞれ control host / worker host の `~/.ssh/authorized_keys` に追記する。

あわせて host key を取り込む。

```bash
ssh-keyscan -H control-node >> ~/.ssh/known_hosts
ssh-keyscan -H worker-node >> ~/.ssh/known_hosts
```

確認:

```bash
ssh -i ~/.ssh/id_ed25519_control controluser@control-node 'hostname'
ssh -i ~/.ssh/id_ed25519_worker1 workeruser@worker-node 'hostname'
```

## 6. Relay 設定を作る

フル構成の公開用サンプルは [../config/relay.remote.example.toml](../config/relay.remote.example.toml) にある。

これを local 設定としてコピーする。

```bash
cp config/relay.remote.example.toml config/relay.local.toml
```

主に書き換える場所:

- `source_sync.ssh_host`, `ssh_user`, `ssh_key_path`, `remote_path`
- `endpoints.a.*`
- `endpoints.a.tunnel.*`
- `routing.default_worker`
- `workers.worker1.*`
- `workers.worker1.tunnel.*`

ポイント:

- `base_url` は tunnel を使う場合 `127.0.0.1:<local_port>`
- `token_env` は文字列必須
- tunnel がある endpoint は、remote `openclaw.json` から token を取得できる
- tunnel がない endpoint は `token_env` の環境変数が必要

## 7. 初回チェック

```bash
PYTHONPATH=src python3 -m openclaw_relay --config config/relay.local.toml check-config
PYTHONPATH=src python3 -m openclaw_relay --config config/relay.local.toml init-state
PYTHONPATH=src python3 -m openclaw_relay --config config/relay.local.toml run --once
```

この時点でエラーになりやすいもの:

- `known_hosts` 未登録
- SSH 鍵 path 間違い
- `responses` endpoint 未有効
- workspace / outbox path 間違い
- host 名解決失敗

## 8. 疎通テスト

control host の outbox にテスト envelope を 1 件置く。

```json
{
  "schemaVersion": "relay-envelope/v1",
  "taskId": "TASK-TEST-001",
  "turnId": "TURN-001",
  "fromGateway": "ControlGateway",
  "fromAgent": "main",
  "toGateway": "WorkerGateway1",
  "toAgent": "main",
  "intent": "request",
  "body": "Reply with the single word hello.",
  "stateRef": "TASK-TEST-001#rev1",
  "returnSessionKey": "main",
  "approvalRequired": false,
  "ttlSeconds": 1800,
  "idempotencyKey": "test-idem-001",
  "createdAt": "2026-03-09T12:00:00+09:00"
}
```

確認:

```bash
curl -fsS http://127.0.0.1:18080/healthz
curl -fsS http://127.0.0.1:18080/readyz
curl -fsS http://127.0.0.1:18080/api/dashboard | python3 -m json.tool | head -n 60
```

ブラウザ確認:

- 会話画面: `http://127.0.0.1:18080/`
- 監視画面: `http://127.0.0.1:18080/ops`

## よくある詰まり

### `readyz` が `ok` にならない

次を確認する。

- `/etc/hosts` または DNS
- SSH 鍵と `known_hosts`
- control host / worker host の SSH server
- `responses.enabled = true`
- OpenClaw gateway 起動状態

### UI に会話が出ない

- control host の outbox path が合っているか
- `source_sync.remote_path` が合っているか
- worker 側 reply が `DONE` まで進んでいるか
- `inspect --limit 20` で DB 状態を確認する

### `401` / `403` が出る

- 環境変数 token が古いか未設定
- または remote `openclaw.json` の token が古い
- 現行実装は auth failure 後に token cache を捨てるので、次回 retry で追従する
