- `OptionABC001 -> OptionGHI003 -> OptionABC001` の mailbox 経路も live で動作確認済み
- `publish-rabbitmq -> OptionGHI003 -> relay reply queue` の direct worker adapter 経路も live で動作確認済み
- Relay 本体は `openclaw-relay.service` で常駐

## 現在の正規経路

現在の正規経路は次。

1. 人間が `OptionABC001` に依頼する
2. `OptionABC001` が Relay mailbox API に最小 message を送る
3. Relay が RabbitMQ mailbox queue に publish する
4. `OptionDEF002` 側 worker が mailbox を取得して処理する
5. `OptionDEF002` の返答を Relay mailbox に送る
6. `OptionABC001` 側 injector が reply を取り込み human-facing `agent:main:main` session に inject する
7. `OptionABC001` が同じ session 上で人間向けに整理して返す

## 現在の重要ルール

- cross-node 通信は internal subagent dispatch を使わない
- `OptionABC001` が書くのは原則 `from` / `to` / `body`
- metadata は Relay が自動補完する
- 返信確認は human-facing `agent:main:main` session で行う
- `outbox/pending` に送信ファイルが残っていても未送信とは判断しない
- `inbox/processed/done/sent` で返信確認しない

## 運用開始の判断

少数 worker 前提なら、いったん運用に入ってよい状態です。

ただし前提はある。

- 現時点の live worker は `OptionDEF002` と `OptionGHI003` の 2 台
- `OptionABC001` が Relay ルールを守ることが前提
- まだ provider-grade の完成版ではない

## 現在の制約

- source sync は `OptionABC001` 1 系統
- live RabbitMQ worker adapter は現在 `OptionDEF002` と `OptionGHI003`
- `OptionGHI003` は mailbox worker 常駐に加えて、SSH tunnel + direct worker adapter でも参加している
- 会話 UI は Relay DB を主、OpenClaw session transcript を補助として使う
- 古い `DEADLETTER_*` は履歴として DB に残る
- queue depth / unacked / redelivery などの broker メトリクスは Grafana まで未整備

## 過去の失敗履歴の扱い

古い `DEADLETTER_A/B` は履歴です。

- 直近の RabbitMQ 正常系が失敗している意味ではない
- 会話画面では直近の `DEADLETTER` は表示し、古いものだけ既定で隠す
- 必要なときだけ `Show deadletters` で表示する
- `/ops` では deadletter を引き続き確認できる

## 今後

今後は次の順で進める。

### 1. 試験運用

- 実タスクを数件流して様子を見る
- `OptionABC001` が human-facing `agent:main:main` session で cross-node 事実を正しく確認できるか定着させる
- 古い internal subagent dispatch 運用を止める

### 2. 1:N 拡張

- `workers.<name>` を追加して複数 worker へ拡張
- worker ごとの adapter service を常駐化
- LAN bind worker と SSH tunnel worker が混在しても運用できるように整理する
- worker ごとの queue / metrics / alert を増やす

### 3. broker 中心の完成度向上

- queue depth / unacked / redelivery の監視追加
- broker DLQ 運用の明確化
- replay / cleanup / alert の broker 基準整理

### 4. provider-grade への再設計

将来的に worker が増えるなら、引き続き [RabbitMQ移行設計.md](RabbitMQ移行設計.md) に沿って v2 を詰める。

特に重要なのは次。

- `accepted` と `completed` の分離
- broker ack と business success の分離
- metadata 中心の routing / retry / replay
- queue を worker 単位に分離したまま拡張すること
