# 監視とアラート

## 目的

Relay の障害を、`systemd` のプロセス死活だけでなく、配送状態、worker ごとの retry / deadletter、latency まで含めて見えるようにする。

## エンドポイント

- `http://127.0.0.1:18080/healthz`
  - プロセス生存確認
- `http://127.0.0.1:18080/readyz`
  - source sync、DB、SSH tunnel、HTTP endpoint を含む準備完了確認
- `http://127.0.0.1:18080/metrics`
  - Prometheus 形式のメトリクス

確認例:

```bash
curl -fsS http://127.0.0.1:18080/healthz
curl -fsS http://127.0.0.1:18080/readyz
curl -fsS http://127.0.0.1:18080/metrics
curl -fsS http://127.0.0.1:18080/metrics | grep -E 'openclaw_relay_worker_(attempt|deadletter|latency)'
```

## 主なメトリクス

- `openclaw_relay_up`
  - プロセスが動いていれば `1`
- `openclaw_relay_ready`
  - Relay が処理可能なら `1`
- `openclaw_relay_source_sync_healthy`
  - `source_sync` が健全なら `1`
- `openclaw_relay_endpoint_tunnel_healthy{endpoint="a|b|..."}`
  - 各 endpoint の SSH tunnel が健全なら `1`
- `openclaw_relay_endpoint_http_healthy{endpoint="a|b|..."}`
  - 各 endpoint の HTTP health が健全なら `1`
- `openclaw_relay_message_status_total{status="..."}`
  - 全体の `DONE`, `FAILED_B`, `FAILED_A_INJECTION`, `DEADLETTER_B`, `DEADLETTER_A` など
- `openclaw_relay_worker_message_status_total{worker="...",status="..."}`
  - worker ごとの status
- `openclaw_relay_attempt_total{target="A|B",result="FAILED|SUCCESS"}`
  - 全体の配送 / 再注入試行回数
- `openclaw_relay_worker_attempt_total{worker="...",target="A|B",result="FAILED|SUCCESS"}`
  - worker ごとの retry / success 累積
- `openclaw_relay_worker_deadletter_total{worker="...",target="A|B"}`
  - worker ごとの deadletter 累積
- `openclaw_relay_worker_latency_sample_count{worker="...",stage="dispatch|inject"}`
  - latency 集計に使われた件数
- `openclaw_relay_worker_latency_seconds{worker="...",stage="dispatch|inject",stat="avg|latest|max"}`
  - worker ごとの dispatch / inject 所要秒数
- `openclaw_relay_watch_pending_files`
  - watch dir 内で、まだ terminal に処理されていない `.json` 数
- `openclaw_relay_watch_present_files`
  - watch dir に物理的に存在している `.json` 数
- `openclaw_relay_spool_files{kind="processing|archive|deadletter|replies"}`
  - spool 各領域のファイル数

## 推奨アラート基準

Prometheus ルール雛形は [../deploy/monitoring/openclaw-relay-alerts.yml](../deploy/monitoring/openclaw-relay-alerts.yml) に置いている。

設計方針:

- `deadletter` と `attempt` は「現在値」より「直近 15 分の増加」で見る
- 今の実環境には既存の `DEADLETTER_B=1` があるため、単純な `> 0` は常時アラートになりやすい
- latency は `latest` を見て、直近悪化を検知する
- pending backlog は 5 秒 polling 前提なので、瞬間値ではなく継続時間で判定する

### critical

- `openclaw_relay_up == 0` が 1 分継続
- `openclaw_relay_ready == 0` が 2 分継続
- `openclaw_relay_source_sync_enabled == 1` かつ `openclaw_relay_source_sync_healthy == 0` が 3 分継続
- `openclaw_relay_endpoint_tunnel_healthy == 0` が 3 分継続
- `openclaw_relay_endpoint_http_healthy == 0` が 3 分継続
- `clamp_min(delta(openclaw_relay_worker_deadletter_total[15m]), 0) > 0`
- `openclaw_relay_watch_pending_files >= 5` が 5 分継続

### warning

- `clamp_min(delta(openclaw_relay_worker_attempt_total{target="B",result="FAILED"}[15m]), 0) >= 3`
- `clamp_min(delta(openclaw_relay_worker_attempt_total{target="A",result="FAILED"}[15m]), 0) >= 3`
- `openclaw_relay_watch_pending_files > 0` が 10 分継続
- `openclaw_relay_worker_latency_seconds{stage="dispatch",stat="latest"} > 120` が 10 分継続
- `openclaw_relay_worker_latency_seconds{stage="inject",stat="latest"} > 30` が 10 分継続

## 現在の運用目安

2026-03-09 JST 時点の実測:

- `OptionDEF002 dispatch avg`: 約 `33s`
- `OptionDEF002 dispatch latest`: 約 `89s`
- `OptionDEF002 inject avg`: 約 `5s`
- `OptionDEF002 inject latest`: 約 `7s`

このため、初期閾値は次で置く。

- `dispatch latest > 120s`
  - warning
- `inject latest > 30s`
  - warning
- backlog `> 0` が 10 分
  - warning
- backlog `>= 5` が 5 分
  - critical

worker 追加や処理内容変更で基準が変わるなら、ここを見直す。

## 障害時の確認順

1. `systemctl status openclaw-relay.service`
2. `curl -fsS http://127.0.0.1:18080/healthz`
3. `curl -fsS http://127.0.0.1:18080/readyz`
4. `curl -fsS http://127.0.0.1:18080/metrics`
5. worker 別メトリクスを grep
6. `inspect --worker ...` で対象 worker のメッセージを確認
7. 必要なら `deadletters --worker ...`
8. 必要なら `replay-deadletter --worker ... --latest`

## 典型コマンド

```bash
sudo systemctl status openclaw-relay.service --no-pager
```
