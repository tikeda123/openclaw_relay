# シンプルメッセージ移行

## 方針

Relay が扱う単位はすべて `message` に統一する。

- 指示
- 依頼
- 進捗
- 完了報告
- 質問
- 返答

を同じ transport で流す。`task` は別 transport として持たず、必要なら本文または metadata に含める。

## なぜそうするか

これまでの構成では、次の 2 系統が混在していた。

- Relay outbox 経由の配送
- OpenClaw 内部 subagent dispatch

この二重化により、会話 UI で

- 見える / 見えない
- Relay を通った / 通っていない
- audit にある / session にしかない

が混ざり、運用判断が難しくなっていた。

## 今後のルール

cross-node の通信はすべて Relay を通す。

- `OptionABC001 -> worker`
- `worker -> OptionABC001`

の両方を Relay の message transport に統一する。OpenClaw 内部 subagent dispatch は、同一ノード内の補助用途に限定し、cross-node 通信には使わない。

## schema

新規導入では `relay-message/v1` を使う。

最小フィールド:

- `schemaVersion`
- `conversationId`
- `messageId`
- `from`
- `to`
- `body`
- `createdAt`

任意フィールド:

- `returnSessionKey`
- `idempotencyKey`
- `ttlSeconds`
- `fromAgent`
- `toAgent`
- `intent`

## サンプル

```json
{
  "from": "OptionABC001",
  "to": "OptionDEF002",
  "body": "Reply with the single word hello."
}
```

Relay が自動で補完する:

- `schemaVersion`
- `from`
- `messageId`
- `conversationId`
- `idempotencyKey`
- `ttlSeconds`
- `createdAt`

互換:

- Relay は `fromGateway` / `toGateway` も引き続き受理する
- ただし新規導入、運用説明、公開 API では `from` / `to` を正とする

`returnSessionKey` を省略するのを推奨する。  
省略時は `OptionABC001` の `default_session_key` を使う。現在の既定は human-facing `agent:main:main` で、互換のため `main` を指定した場合も `agent:main:main` に正規化する。

## 互換

既存の `relay-envelope/v1` は当面受理する。  
ただし、運用・文書・サンプル設定は `relay-message/v1` を正とする。

互換の意味:

- DB 内部の `task_id` / `turn_id` は当面残る
- 画面や外向け説明では `conversationId` / `messageId` を優先する

## 実装上のゴール

1. `OptionABC001` が cross-node 通信を outbox message に統一する
2. Relay UI は message stream を主表示にする
3. audit / metrics / deadletter も message 単位で統一する
4. session transcript 依存は補助扱いに下げる
